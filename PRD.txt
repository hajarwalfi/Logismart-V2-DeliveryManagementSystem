Application de gestion des livraisons avec Spring Boot, pour suivre, planifier et optimiser l’envoi de colis.
Référentiels
Compétences transversales
[2023] Concepteur⋅rice développeur⋅se d'applications
Contexte du projet
La société SmartLogi, spécialisée dans la livraison de colis à travers le Maroc, souhaite moderniser et automatiser la gestion de ses opérations logistiques.

Actuellement, la société gère les livraisons manuellement à l’aide de fichiers Excel et de registres papier, ce qui engendre plusieurs difficultés : erreurs de saisie, pertes de données, retards dans les livraisons et manque de visibilité sur le suivi des colis.

Pour répondre à ce besoin, votre société de services en développement logiciel a été sollicitée pour concevoir et développer une solution web de gestion logistique, nommée Smart Delivery Management System (SDMS).

Cette application devra permettre à SmartLogi de :

Collecter les colis depuis les clients expéditeurs.
Stocker temporairement les colis dans les entrepôts de la société avant expédition.
Planifier et assurer la livraison des colis aux destinataires finaux, qu’ils soient dans la même ville ou dans une autre région du Maroc.
Fournir une traçabilité complète des colis grâce à un historique des statuts.
Optimiser la planification logistique par zones géographiques et selon la priorité des colis.
Réduire les erreurs humaines et améliorer la fiabilité du suivi des livraisons.
Votre mission consiste donc à développer la nouvelle version du SDMS, plus complète et évolutive, qui servira de base au futur système intégré de gestion logistique de SmartLogi.

Objectifs
Centraliser la gestion des clients expéditeurs, destinataires, colis, livreurs et zones de livraison.
Suivre le flux complet des colis : collecte chez le client, stockage, planification de la tournée, livraison finale.
Permettre une recherche et un suivi précis des colis via différents critères (ville, zone, statut, priorité).
Fournir un historique complet de chaque colis pour le suivi et la traçabilité.
Optimiser l’organisation des tournées et réduire les retards grâce à une planification efficace.
Faciliter la prise de décision pour les gestionnaires logistiques.
Utilisateurs
Gestionnaire logistique : supervise toutes les opérations, planifie les tournées, gère les livreurs et le stock.
Livreur : consulte ses colis assignés et suit sa tournée.
Client expéditeur : crée des demandes de livraison et reçoit des notifications sur l’état de ses colis.
Destinataire : reçoit le colis et peut suivre sa livraison.

User Stories

L’entreprise souhaite désormais sécuriser son API REST afin de :

protéger les données sensibles,
séparer clairement les droits entre gestionnaires, livreurs et clients,
préparer l’intégration avec des frontends Angular/React,

Référentiels
Compétences transversales
[2023] Concepteur⋅rice développeur⋅se d'applications
Contexte du projet
Contexte du projet
L’application SmartLogi Delivery Management System (SDMS) est une solution de gestion dédiée aux activités de livraison, utilisée par différents profils d’utilisateurs tels que les gestionnaires, les livreurs et les clients expéditeurs.

Avec la croissance de l’activité et l’augmentation du nombre d’utilisateurs, SmartLogi souhaite faire évoluer les modalités d’accès à l’application afin d’améliorer l’expérience utilisateur et de s’aligner sur les usages des applications modernes. Certains utilisateurs privilégient une connexion avec des identifiants classiques, tandis que d’autres souhaitent utiliser des comptes existants largement répandus afin de faciliter et d’accélérer l’accès aux services, sans contrainte de création d’un nouveau compte spécifique.

Dans ce contexte, l’objectif est d’enrichir les modes d’accès à SDMS tout en conservant une gestion cohérente et centralisée des comptes utilisateurs. Cette évolution vise à :

proposer plusieurs modes de connexion adaptés aux différents profils et usages ;
garantir une expérience d’authentification fluide, simple et sécurisée ;
maintenir une gestion unifiée des droits d’accès aux fonctionnalités métier, indépendamment du mode de connexion choisi ;
préparer l’application à une utilisation à grande échelle, notamment via des interfaces web et mobiles modernes.
Cette évolution s’inscrit dans une démarche d’amélioration continue de la plateforme, en assurant fiabilité, sécurité et facilité d’usage, sans remettre en cause les mécanismes existants.

Objectifs du projet
L’objectif est de mettre en place une authentification hybride :

Authentification classique : email / mot de passe → JWT
Authentification OAuth2 : Google / Apple / Facebook → JWT interne généré par l’application, avec possibilité de s’appuyer sur un serveur d’identité externe (Okta).
Ainsi, quelle que soit la méthode d’authentification utilisée, l’API SDMS continue de fonctionner de manière 100 % stateless, en s’appuyant exclusivement sur des Bearer JWT pour sécuriser les endpoints.

Les objectifs spécifiques sont les suivants :

ajouter une authentification OAuth2 avec des fournisseurs sociaux (Google, Apple, Facebook) ;
conserver et réutiliser le mécanisme existant basé sur Spring Security et JWT stateless ;
enregistrer ou mettre à jour les informations des utilisateurs OAuth2 en base de données ;
générer un JWT interne identique à celui émis lors d’une authentification classique ;
centraliser la sécurité de l’API autour des Bearer JWT ;
préparer l’application à une intégration fluide avec des frontends Angular et React ;
déployer l’application dans un environnement Docker multi-conteneurs.
Utilisateurs concernés
Gestionnaire logistique
Livreur
Client expéditeur
Les rôles et permissions existants restent inchangés. Un utilisateur authentifié via OAuth2 se verra attribuer un rôle par défaut (ex. : CLIENT), modifiable ultérieurement par un administrateur.

Contexte du projet
Contexte métier
L’application SmartLogi Delivery Management System (SDMS) a connu plusieurs évolutions majeures :

v0.1.0 : fonctionnalités métier (gestion des colis, clients, livreurs, zones)
v0.1.1 : qualité logicielle et tests automatisés
v0.2.0 : sécurisation JWT stateless
v0.2.1 : authentification hybride JWT + OAuth2 et déploiement Docker
À ce stade, l’application est fonctionnelle, sécurisée et conteneurisée, mais le cycle de développement repose encore sur des actions manuelles (tests, build, analyse, déploiement).

Dans un contexte professionnel réel, cette approche engendre :
des risques d’erreurs humaines,
des délais de mise en production,
une difficulté à garantir la qualité continue du code.
SmartLogi souhaite donc industrialiser le cycle de vie de l’application en mettant en place une chaîne CI/CD complète, permettant :

une validation automatique du code à chaque modification,
une détection précoce des régressions et problèmes de qualité,
une standardisation des builds et des déploiements,
une meilleure collaboration entre les développeurs.
Objectifs métier
L’objectif de cette itération est de sécuriser et fiabiliser le processus de livraison logicielle du SDMS.

À l’issue de cette phase, SmartLogi doit être capable de :

intégrer automatiquement les nouvelles fonctionnalités développées,
s’assurer que chaque changement respecte les règles de qualité et de sécurité,
livrer rapidement une version exploitable de l’application,
disposer d’indicateurs clairs sur l’état de santé du projet.
Cette démarche vise à préparer le SDMS à un déploiement en environnement professionnel (préproduction / production).

Périmètre fonctionnel
La chaîne CI/CD devra couvrir l’ensemble du cycle suivant :

Développement
Un développeur pousse du code sur le dépôt Git (GitHub).
Validation automatique
Le code est automatiquement compilé.
Les tests unitaires et d’intégration sont exécutés.
La qualité du code est analysée.
Contrôle qualité
Les anomalies, vulnérabilités et dettes techniques sont détectées.
Le projet est bloqué si les critères de qualité ne sont pas respectés.
Livraison
Une image Docker de l’application est générée.
L’application est prête à être déployée de manière standardisée.
Spécifications techniques
Outils et technologies
La mise en place du CI/CD devra s’appuyer sur les outils suivants :

Gestion de code source : Git/GitHub (branches, pull requests)
Intégration continue : Jenkins
Build & Tests: Maven, JUnit 5, Mockito
Qualité du code: SonarQube, JaCoCo
Conteneurisation : Docker, Docker Compose
Bonnes pratiques CI/CD
Utilisation d’un pipeline déclaratif Jenkins.
Séparation claire des étapes du pipeline.
Variables d’environnement pour :
accès SonarQube,
configuration Docker.
Logs lisibles et exploitables.
Pipeline reproductible sur un autre environnement.
Évolutions optionnelles (bonus) – Déploiement Cloud (AWS)
Dans une optique de mise en situation professionnelle, une extension optionnelle du projet consiste à déployer l’application SDMS sur une infrastructure cloud AWS, en s’appuyant sur les principes DevOps et CI/CD déjà mis en place.
